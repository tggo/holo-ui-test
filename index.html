Holo 3.1vsClaude

A UI test needs two roles — eyes that ground a screenshot into a click, and a brain that decides the next move. We swap who plays each role and run the same 9-action checkout, measuring wall-time, tokens and cost.

Holo 3.1 35B-A3B Q4 · llama.cpp qwen2.5:7b · ollama Sonnet 4.6 Playwright · Chromium

the split

Eyes & brain

Holo is a grounding model — it turns “find the Cart link” into pixels. The reasoning can live in a separate LLM. That lets the brain be small and local while Holo handles sight.

⌁ Brain — planning

how a run works

The test loop

One step = the brain reads the page and picks an action, the eyes turn that into a click, the browser updates. Repeat until the brain says done. Swap either model without touching the loop.

local brain placement

GPU vs CPU

Holo’s 21 GB sit on the Metal GPU. A co-resident brain there overflows the default working-set (~27 GB on a 36 GB Mac) → Metal OOM. Raise iogpu.wired_limit_mb to 31000 and the brain fits.

The variance was Spotlight. mds_stores re-indexing the code volume sat at 76% CPU and doubled the run. Quiet (mdutil -a -i off) + GPU, the fully-local config drops to 46 s — faster than Holo+Sonnet (58 s) and within ~6 s of all-cloud. Progression: 98 → 89 → 60 → 46 s. The eyes still dominate — Holo grounds in ~5 s/call vs Sonnet’s ~2 s.

when to pick what

Verdict

latency matters

Sonnet + Sonnet

Fastest (40 s), simplest, one vendor. The default when a key is fine and screenshots may leave the box.

on-box / $0

Holo + qwen2.5:7b

Fully local, zero marginal cost, nothing leaves the machine. Pays ~2× the wall-time for it.

niche

Holo + Sonnet

Buys little — you still pay Sonnet for the brain and Holo’s eyes are slower. Only if the brain must be frontier and pixels can’t leave.

field notes

What actually broke

  • 01Holo won’t run on ollama. Its engine doesn’t implement the qwen35moe arch and the hf.co pull 400s on the manifest. Served the cached GGUF + vision projector directly with llama.cpp instead.
  • 02The brain is the fragile part, not the eyes. A 3B (llama3.2) loops and emits click|type; qwen2.5:7b is the floor that plans this flow. Holo & Sonnet eyes ground to near-identical pixels.
  • 03Disambiguate repeated controls. Four identical “Add to cart” buttons — the observation tags each with its card text (in: Mechanical Keyboard $89) so the brain can target the right one.
  • 04Parse JSON with raw_decode. Both Claude and small local models append prose after the object; grabbing the first complete object from the first { survives it.

reproduce

Run it yourself

Two model servers (Holo on llama.cpp, the brain on Ollama) + a Playwright harness. macOS / Apple-Silicon shown; a 36 GB Mac runs the 35B-A3B Q4 + a 7B brain.

1 · eyes — Holo 3.1 via llama.cpp (:8080)
# ollama can't run the qwen35moe arch — serve the GGUF + vision projector directly
brew install llama.cpp
llama-server -hf Hcompany/Holo-3.1-35B-A3B-GGUF:Q4_K_M \
  --alias holo31 --host 127.0.0.1 --port 8080 \
  -c 4096 -ngl 999 -fa on --jinja
2 · brain — qwen2.5:7b via Ollama (:11434)
# use the official build from ollama.com (the Homebrew bottle ships no GGUF runner)
ollama pull qwen2.5:7b
3 · mac — let the brain share the GPU
# Holo's 21 GB + a 7B brain exceed the default Metal working-set → raise it
sudo sysctl iogpu.wired_limit_mb=31000
# …or skip this and run the brain on CPU (default, no flag needed)
4 · harness — setup & run
python3 -m venv .venv && . .venv/bin/activate
pip install openai anthropic playwright pydantic
playwright install chromium
echo 'ANTHROPIC_API_KEY=sk-ant-…' > .env       # for the Sonnet roles

# all three configs, watch the browser
python bench.py --scenario checkout-flow --headed

# fully-local only, brain on the GPU
python bench.py --only "Holo + LocalModel" --brain-gpu
the whole idea, in a snippet
def step(page, goal, history):
    # BRAIN: page text + goal + history → next action
    dec = brain.next_action(goal, observe(page), history)
    # → {"action":"click","target":"Add to cart for Mechanical Keyboard"}

    # EYES: screenshot + that target → pixel coordinates
    hit = eyes.locate(page.screenshot(), dec.target, 1280, 800)
    page.mouse.click(hit.px, hit.py)        # execute, observe, repeat
